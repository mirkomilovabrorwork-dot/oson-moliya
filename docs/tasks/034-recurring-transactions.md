# Task 034 — Recurring transactions (IMPLEMENTATION)

**Status:** SPEC · 2026-06-18 · Opus (autopilot, full autonomy granted by user)
**Owner agent:** Sonnet (`claude-sonnet-4-6`)

## 1. Goal

Real SMB use: rent every 1st (Ijara 2M), salary every 25th (Ish haqi 15M), monthly bills, quarterly
tax. Today these are entered manually each month. Add `RecurringRule` so the user defines the
template ONCE; Vercel Cron creates the actual `Transaction` rows on schedule.

## 2. Decisions locked (Opus chose, autopilot autonomy)

- **D1. Generation = Vercel Cron at midnight Tashkent (UTC+5).** Cron path `/api/cron/recurring`,
  protected by `CRON_SECRET` Bearer header. Simple, reliable, no on-load surprises, no bot prompts.
- **D2. Schedule = monthly + yearly only.** `dayOfMonth` 1..28 (avoid Feb 29-31 edge cases). For
  yearly add `monthOfYear` 1..12.
- **D3. Past frozen, future stops.** Editing or deleting a rule does NOT change already-generated
  transactions. Delete = soft (`deletedAt`); generator skips deleted rules.
- **D4. Category-deletion handling.** If a rule's category is deleted → set `categoryId = null`
  (DB-level via `onDelete: SetNull`), pause the rule (`pausedAt = now()`), and show a "Needs
  category" badge in the recurring-rules UI. Generator skips paused rules until user fixes.
- **D5. One currency per rule.** Multi-currency = multi-rule. Simple.

Additional decisions:
- Generator is **idempotent**: tracks `lastGeneratedAt`. On each run, for each active rule, computes
  the next "due date" (= the next occurrence ≥ `max(startDate, lastGeneratedAt + 1 period)` ≤ today),
  and generates a Transaction at that date. If the cron missed a day (server outage), it CATCHES UP
  by generating ALL missed occurrences in one run.
- Rule's catch-up window: never go further back than the rule's `startDate`. Never go past `endDate`
  if set.
- Transactions generated by recurring carry `source = "recurring"` (distinguishable from `bot` / `web`).

## 4. Schema (additive only)

```prisma
enum RecurringFrequency {
  monthly
  yearly
}

model RecurringRule {
  id               String             @id @default(cuid())
  userId           String
  user             User               @relation(fields: [userId], references: [id], onDelete: Cascade)
  type             TxType
  categoryId       String?
  category         Category?          @relation(fields: [categoryId], references: [id], onDelete: SetNull)
  amountUzs        BigInt
  originalCurrency String?
  originalAmount   BigInt?
  note             String?
  frequency        RecurringFrequency
  dayOfMonth       Int                // 1..28
  monthOfYear      Int?               // 1..12 for yearly, null for monthly
  startDate        DateTime
  endDate          DateTime?
  pausedAt         DateTime?
  lastGeneratedAt  DateTime?
  deletedAt        DateTime?
  createdAt        DateTime           @default(now())

  generatedTransactions Transaction[] @relation("RecurringRuleTransactions")

  @@index([userId, deletedAt])
}
```

## 5. Files to touch / create

1. **`prisma/schema.prisma`** — additions above.

2. **`src/lib/services/recurring.ts`** (NEW) —
   - `listActiveRules(userId)` → rules where `deletedAt: null`, ordered by `startDate`.
   - `createRule(input)` → validates type+amount+frequency+date fields; for `monthly`, requires
     `dayOfMonth` in [1, 28]; for `yearly`, requires both `dayOfMonth` and `monthOfYear`.
   - `updateRule(id, userId, patch)` → owner-scoped; same validations.
   - `deleteRule(id, userId)` → soft delete (set `deletedAt`).
   - `pauseRule(id, userId)` / `resumeRule(id, userId)` → toggle `pausedAt`.
   - **`generateDueTransactions(asOfDate?: Date)`** → the cron worker.
     - For EACH user-rule where `deletedAt:null AND pausedAt:null AND categoryId IS NOT NULL`:
       - Compute the next due-date based on `frequency`, `dayOfMonth`, `monthOfYear`, `startDate`,
         and `lastGeneratedAt`:
         - If `lastGeneratedAt` is null → first due-date = the first occurrence on/after `startDate`.
         - Else → next occurrence is `lastGeneratedAt + 1 month` (monthly) or `+ 1 year` (yearly),
           clamped to the schedule's `dayOfMonth`.
       - **Catch up loop:** while `nextDue ≤ asOfDate AND (endDate IS NULL OR nextDue ≤ endDate)`:
         - Create a Transaction with the rule's amount/type/category/note/originalCurrency, set
           `source = "recurring"`, `recurringRuleId = rule.id`, `occurredAt = nextDue`,
           `userId = rule.userId`.
         - Update `lastGeneratedAt = nextDue`.
         - Compute next iteration `nextDue`.
     - Wrap each rule's catchup in try/catch; failures on one rule must not stop others.
     - Return `{ rulesProcessed, transactionsCreated, errors: [{ ruleId, message }] }`.
   - **All date math uses Tashkent timezone helpers from `src/lib/dates.ts`** — never raw UTC.
